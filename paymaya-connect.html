          PayMaya Connect for Third Party Integration

          PayMaya Connect follows the OpenID Connect standard to provide third parties a familiar way to integrate with PayMaya APIs.

          Terminology ¶

          Term Description
          PayMaya Connect The OpenID Provider/Authorization Server
          Client ID Client identifier as registered or generated by PayMaya Connect
          Client Secret Client secret as generated by PayMaya Connect
          Access Token Token in JWT format used to access protected resources
          Refresh Token Token in JWT format used to retrieve new access token without having to ask the user to re-authenticate/authorize
          ID Token Token in JWT format used to verify user authentication
          Scope Permission the user must authorize to access resources

          Authorization Code Grant ¶

          To retrieve an access token with access on behalf of a user:

          Step 1 ¶

          Redirect the user to http://accounts.paymaya.com/authorize, along with the following query parameters:

          Query Param Description
          scope OPTIONAL. Space-delimited list of scopes you want to request the user to authorize. Omitting this will default to requesting all the scopes the client can have according to client registration.
          response_type REQUIRED. Response type to indicate the authorization processing flow to be used. When using the Authorization Code Flow, this value is code.
          client_id REQUIRED. Client identifier as registered or generated from PayMaya Connect (OpenID Provider/Authorization Server).
          redirect_uri REQUIRED. Redirection URI to which the response will be sent. This URI MUST exactly match one of the Redirection URI values for the Client pre-registered at the OpenID Provider
          state RECOMMENDED. Opaque value used to maintain state between the request and the callback. Typically, Cross-Site Request Forgery (CSRF, XSRF) mitigation is done by cryptographically binding the value of this parameter with a browser cookie.
          prompt OPTIONAL. May be set to login, none, or be omitted. If set to login, any user session will be destroyed and user will be asked to re-authenticate. If set to none, user will not be prompted for any authentication or consent pages. If user has an active session, PayMaya Connect will redirect user to client successfully. If user has no session this will return an error query parameter with login_required as value in the redirection.
          user_id OPTIONAL. User identifier such as email or mobile number with which to pre-fill the login form.

          Example:

          https://accounts.paymaya.com/authorize?
    client_id=6e1a704b-0eba-4ff0-89d5-ade9762e2705&
    response_type=code&
    redirect_uri=https://client.com/callback
    &state=1234

          Once the user authorizes, he will be redirected back to the specified redirect_uri, along with the following:

          Parameter Description
          code Generated authorization code. This is exchanged with an access token.
          state The same state value as supplied in the authorization request.
          error Added if there is any error. e.g. login_required

          Example:

          https://client.com/callback?code=X2wy4L&state=1234

          grant_type Use authorization_code
          code The authorization code received from the prior request
          redirect_uri Identical redirect_uri as the prior request

          Client Authentication ¶

          There are different authentication options for clients to use to request for access token. The authentication method will default to client_secret_basic if unspecified during registration.

          client_secret_basic ¶

          Include the client credentials using Basic Auth where username is the client_id and the password is the client_secret

          client_secret_post ¶

          Include the client credentials in the request body using the ff parameters:

          Parameter Description
          client_id The client ID
          client_secret The client secret

          client_secret_jwt ¶

          Use a JWT signed by the client secret to do client assertion. Include the following parameters in the request body:

          Parameter Description
          client_id The client ID
          client_assertion_type Use urn:ietf:params:oauth:client-assertion-type:jwt-bearer
          client_assertion JWT signed using the client secret. Refer below for required claims.

          private_key_jwt ¶

          Use a JWT signed by a private key to do client assertion. For the assertion to be verified, either provide the uri to the public key using a jwks_uri or provide the public key json itself in jwks during client registration.

          Parameter Description
          client_id The client ID
          client_assertion_type Use urn:ietf:params:oauth:client-assertion-type:jwt-bearer
          client_assertion JWT signed using a private key. Refer below for required claims.

          JWT-based Authentication Required Claims ¶

          Claim Description
          iss REQUIRED. Issuer. This MUST contain the client_id of the OAuth Client.
          sub REQUIRED. Subject. This MUST contain the client_id of the OAuth Client.
          aud REQUIRED. Audience. The aud (audience) Claim. Value that identifies the Authorization Server as an intended audience. The Authorization Server MUST verify that it is an intended audience for the token. The Audience SHOULD be the URL of the Authorization Server’s Token Endpoint.
          jti REQUIRED. JWT ID. A unique identifier for the token, which can be used to prevent reuse of the token. These tokens MUST only be used once, unless conditions for reuse were negotiated between the parties; any such negotiation is beyond the scope of this specification.
          exp REQUIRED. Expiration time on or after which the ID Token MUST NOT be accepted for processing.
          iat OPTIONAL. Time at which the JWT was issued.

          Scopes ¶

          The following are the available scopes. Note that not specifying a scope during

          Scope Description
          openid To additionally return ID Token when retrieving access token
          offline_access To additionally return Refresh Token when retrieving access token
          alias To generate transfer codes
          bill To pay bill
          transfer To send money
          balance To view balance
